# C-8 設計案: 直接盤面合成（現在盤面の要約制約サンプリング）

Status: reviewed（2026-07-21 codex 設計レビュー済み・指摘反映済み。要点は末尾）

## 背景と狙い

C-7（`docs/c7-continuous-filter.md`）は粒子フィルタを連続重み化し、窓付き
若返り・墓場復活・ε_phys の最後の砦まで積んだが、kakunari の c42
（8七の三重取り合いの後、相手飛車が観測ゼロのまま数手かけて8筋へ回る
「向かい飛車」の多手サイレント再配置）は解けないまま残った。原因は構造的:
現行フィルタは「観測を**全部説明する指し手列**」を探しており、これは
仮説空間が着手ごとに掛け算で爆発する（kakunari 70手目時点で概算 30^35）。
針の穴を何十個も連続で通す設計になっている。

一方、実測でわかったことが2つある:

1. **taint 粒子（ε_phys の最後の砦、盤面は部分的に嘘）でも玉位置の信念は
   91.8% で真実に一致する**（scenario kakunari diag）。局所の駒種・経路は
   崩れても、大域情報は残る
2. kakunari を最初に分析したときのユーザー（有段者）の信念記述は、指し手列の
   列挙ではなく**要約された制約**だった: 「相手玉が４七/３七に集中」
   「５七への相手利き≥2枚の確率が低い（相手の手が全部攻めでテンポを守りに
   使っていない、自分の４八歩打→４九歩成が通ったことが『４八に金銀なし』を
   証明する負の証拠）」

C-8 は「指し手列を探す」のをやめて、**現在の盤面そのものを、履歴を再現せず
要約制約から直接サンプルする**方向への転換案。人間の推論様式（テンポ収支・
負の証拠・駒の勘定）に構造を寄せる。

## 使える「無料」の情報（履歴再現なしで手に入るもの）

`GameModel`（model.rs）と `Estimator` の既存トラッキングから、以下は
**手順の再現なしに厳密に分かる**:

- **相手の持ち駒**: `GameModel::opponent_hand()`（取られた自駒の成りを
  戻したもの。完全に既知）
- **相手の盤上駒数**: 20 − 自分が取った枚数（`strategy.rs` の `opp_board_n`
  で既に使用中）
- **相手の盤上駒の役割別内訳**: 初期20枚 − 取られた駒種 − 相手持ち駒中の
  同役割（駒の入れ替わりは役割単位で管理すればよい。個体識別は将棋の駒に
  意味がないので不要）
- **自分が駒を取ったマス・取られたマス**: 既存の `my_capture_sq` /
  `lost_pieces()` で既知
- **玉位置の信念**: taint 粒子由来（91.8%）。C-8 の合成でも種として使える

## 設計の核（3本柱）+ 1つの重要な訂正

### D1: テンポ収支（tempo budget）

相手の総手数 N（`GameModel::opponent_moves()`、厳密に既知）に対し、現在の
相手配置を作るのに必要な最小手数の合計は N を超えられない。これは chess の
distance-heuristic と同型で、Shogi では駒の分類が2種類必要:

- **「生え抜き」駒**（まだ一度も取られていない初期20枚のうち現存するもの）:
  初期位置からの最短到達手数（盤上を空とみなした admissible な下限。
  桂は跳躍グラフの最短路、飛車角香は directions のグラフ距離、歩は
  前進段数、金銀は最短路）
- **「持ち駒復帰」駒**（自分が取られた駒が相手に取られ→相手が打った駒。
  `lost_pieces()` に対応）: 未成のまま盤上にあるなら打った1手だけがコスト。
  **成り駒の場合は打つ1手 + 打ち先から現在地までの最短移動手数が必要**
  （codex 指摘: 打った瞬間は生駒でしか存在できないので、成り駒として
  発見された持ち駒復帰駒は追加の移動コストを払っている）

**割り当て問題であることに注意**（codex 最重要指摘の一つ）: 盤上に同役割の
駒が複数ある場合、どれを「生え抜き」・どれを「持ち駒復帰」とみなすかで
最小テンポの値が変わる。単純な役割多重集合の合算では過小/過大評価になり、
役割ごとの**小さい二部マッチング**（生え抜き候補×持ち駒復帰候補の割り当て
コスト最小化）を解く必要がある。貪欲割り当ては、緩すぎて偽の配置を通す
方向にも、厳しすぎて真実を棄却する方向にも転びうる。

`min_assignment_cost(生え抜き候補, 持ち駒復帰候補) ≤ N` が合成した配置が
満たすべき制約（下限のみを課す不等式なので、緩めに使う分には安全側）。

**成りの経路制約**（codex 指摘）: 「成りは移動に付随するフラグで手数を
消費しない」だけでは不十分。成り駒として配置する場合、**成れるゾーンを
通過可能な最短路**になっているか（生駒のグラフ距離だけでなく、経路が
成れる段に到達するか）を検証する必要がある。

### D2: 駒の勘定（既存の情報を配置制約として明示）

盤上に置く駒の多重集合は「相手の役割別内訳」（上記）から決まる。持ち駒は
`opponent_hand()` からそのまま。二歩・行き所のない駒（強制成り）などの
将棋の配置合法性（`shogi.rs` の `is_pseudo_legal` が使っている
`has_pawn_on_file` / `dead_end_rank`）は、合成時にも独立に守る必要がある
（これは今まで「指し手の合法性」経由で暗黙に守られていたが、配置を直接
サンプルするなら明示的な配置制約として実装しなければならない）。

**打ち歩詰めは配置レベルでは原則チェック不能**（codex 指摘）: 打ち歩詰めは
「直前の手が打ち歩による詰みだったか」という**手**の合法性であり、
現在配置だけからは判定できない（直前の手が打ち歩由来かどうかの情報が
残っていないため）。既知のギャップとして記録し、対応しない（実害は
「打ち歩詰めのはずの配置を許してしまう」という稀なケースの過大許容のみ）。

### D3: 配置事前分布（学習済み）

`likelihood.rs` の8特徴量（king_moved・king_advance・pawn_advance・
pieces_home 等）は、粒子群の中で真の局面を判別する条件付きMLEで学習済み
（`bin/fit_particles`）。現状は「粒子を作ってから再重み付けする」使い方
だが、C-8 では同じ特徴量（または拡張版）を**生成モデルの重み**として
使い直す（既存の再重み付け用フィットとは別に、配置を直接生成する用途向けの
再フィットが必要）。

### 【重要な訂正】負の証拠は永続的な制約ではなく、時間減衰する

当初「自分の手が通った/打てたマスの空き」を**現在も成り立つハード制約**
として扱う想定だったが、これは誤り。私の飛車が時刻Tにあるマスを素通り
できた（=相手駒が無かった）という事実は、**その時刻Tの局面についてしか
証明しない**。時刻Tの後、相手の駒がそのマスへ移動してくる可能性を排除
できないため、「今」の配置制約としては使えない（相手が実際にそこへ来た
なら、それ自体が別の観測で説明されるべきだが、その観測を追跡していない
限りマスが今も空という保証にはならない）。

つまり「負の証拠」は本質的に**直近の観測ほど信頼できるが、古い観測ほど
腐る**という時間窓つきの情報であり、C-7 の窓付き若返り（直近 REJUV_SNAPSHOTS
決定点）と同じ発想の「限定的な履歴」が結局必要になる。純粋な「履歴なしの
スナップショット合成」は成立しない（codex レビューで妥当と確認）。

**C-7 との違いの正確な言い方**（codex 指摘で明確化）: 両者とも「直近の
限定的な履歴」を使う点は共通だが、サンプル対象が違う。

- C-7（windowed rejuvenation）= **path-space sampling**: 「窓内で相手が
  指した手の並び」を当てにいく。中間の指し手ひとつひとつが正しくないと
  失敗するので、多手のサイレント再配置（kakunari c42）に弱い
- C-8 = **state-space synthesis**: 「現在の配置」を直接当てにいく。中間の
  経路（相手飛車がどの順で8筋へ来たか）を当てる必要がない。共通するのは
  「直近観測をソフト制約として使う」点だけ

C-8 の構造的優位性はここに限定される: **中間経路を当てなくてよいこと**。
配置同士の相関（複数駒が連携して利きを作っている、等）を表現できるかは
別問題で、後述のとおり案Aの独立成分近似はここが弱い（主要リスク4）。

## 合成アルゴリズム案（2方式、レビューで比較したい）

### 案A: 構築的サンプリング（ancestral sampling）

1. 玉位置を taint/尤度由来の分布からサンプル
2. 残りの駒を役割ごとに1枚ずつ、実現可能マス（既に埋まっていない・
   二歩/強制成り制約を満たす・直近負の証拠に反しない）から、配置事前分布 ×
   テンポ収支の残り予算に応じた重みでサンプル。置くたびに残りテンポ予算を
   消費（distance 分）
3. 全駒を置き終えたら、テンポ収支の合計 ≤ N を満たしているか確認
   （満たさなければ棄却して最初からやり直す）

長所: 実装・デバッグが単純、暴走しにくい。短所: 独立成分近似なので、
駒同士の相関（利きの形など）を表現しにくい。

### 案B: MCMC（Metropolis-Hastings、局所編集）

現在の（taint でもよい）粒子を種にして、駒1〜2枚の位置を入れ替え/移動する
局所編集を提案し、配置事前分布 + テンポ収支ペナルティのスコア比で
受理/棄却する。C-7 の窓付き若返りが「時系列の巻き戻し」で修復するのに対し、
こちらは「配置の巻き戻し」で修復するイメージ。

長所: 案Aが苦手な駒同士の相関（利きの形）を表現しやすい。短所: 混合の
遅さ・チューニングの手間。案Aの結果を初期値にしてMCMCで仕上げる、という
併用も考えられる。

**現時点の推奨**（codex レビューで修正）: kakunari の needle 自体が
「複数駒の連携」（相手飛車が8筋へ回り、王手の幾何を作る）なので、案A
（独立成分近似）**単体では本丸に刺さらない**可能性が高い。完成形は
**案A（構築的初期化）+ 短い案B（MCMC局所編集での仕上げ）**の組み合わせとし、
P2 の内側で「案Aだけの結果」と「案A→案Bで仕上げた結果」を比較する
（利きの形など駒間相関の指標で測る）。初期実装は案Aから着手するが、
「案A単体で十分」という前提は置かない。

## 生成粒子の身分（provenance）— 要レビューの論点

C-8 が生成する配置は、以下を満たす:
- 持ち駒・盤上駒数・駒の役割内訳（完全一致、ハード制約）
- 二歩・強制成り等の配置合法性（ハード制約）
- テンポ収支の下限（不等式、ハード制約）
- 直近の負の証拠（減衰つきソフト制約）

しかし**厳密整合（info_miss=0）粒子が満たす「観測列全体との逐次整合」は
検証していない**（例: 10手前の王手宣言が要求した幾何が、今の配置と矛盾
しないか、はチェックしない）。したがって現行の「厳密 / ソフト / taint」の
3層のどこにも素直には当てはまらない。

**提案（codex レビューで確定）**: 第4の身分 `synth` を新設し、**厳密粒子とは
同格に扱わない**。P1（logw 統一・ESS リサンプリング）で経験した
prior_weight 較正崩壊（multiplicity 導入時に legacy_mass の定義を誤ると
p(合法) ブレンドが壊れる）と同じ失敗パターンを避けるため、evaluation 側は
最初から4層を明確に分離する:

- `strict`: p_legal・捕獲期待値・リスク・攻め・王手ソルバーにフルに使える
- `info_soft`（既存の info_miss）: 減衰つきで一部使用（既存どおり）
- `synth`（新設）: **用途限定**。初期段階では玉位置分布・利き分布・
  候補手の相対評価にのみ使う。**p_legal のブレンドには混ぜない**
  （`strict_unique_effective_n` とは別勘定の `synth_unique_effective_n` を
  持ち、加算しない — 「独立証拠が増えた」と誤認させないため）
- `phys_taint`（既存）: 原則、玉位置分布のみ（現状どおり）

`synth` が `strict` に昇格できるかどうかは、実測（合成配置が実際どれだけ
「本物」に近いか。P2 の測定指標参照）で判断する。初期実装では昇格させない。

## 統合案（保守的な第一歩）

C-8 をいきなり主要な生成経路（`replay_once` の置き換え）にはせず、まず
**C-7 の墓場復活・若返りが失敗し、ε_phys へ落ちる直前の「もう一段の砦」**
として挿入する。既存の P1〜P3 の枠組みは変えない。C-8 が実用に足ると
分かった段階で、初期リプレイ自体の置き換え（大きい変更、二段目のフェーズ）
を検討する。

ブラインド玉攻め勾配（C-7 P3 追補）は taint の玉位置分布だけを取り出す
狭い成功例だったが、C-8 が実用化すれば同じ発想がもっと広い特徴量
（駒の勘定・利きの形）に及ぶので、ブラインド玉攻めは「C-8 が効かない
ときの保険」として残る（競合ではなく包含関係）。

**統合の絞り込み**（codex レビューで具体化）: `synth` を挿入しても、
最初の統合はかなり狭く絞る:

- クリーン粒子が十分あるとき（`strict_unique_effective_n` が閾値以上）は
  `synth` を一切使わない
- クリーン全滅、または `strict_unique_effective_n` が極小のときだけ発動
- `synth` の総質量に上限を設ける（評価を乗っ取らせない）
- p_legal の事前ブレンドには混ぜない（上記の provenance 節のとおり）
- 採否判定は必ず `synth` 無効（フラグで切れるようにする）との
  match_seed ペア比較で行う

**もう一つの選択肢として並走させる価値がある案**（codex 指摘、C-8 の
代替/補完）: **多段ガイド**（C-7 の Guide を1手の状態条件から複数手の
依存チェーンへ拡張する）。「将来 X に飛車が必要」という結論から逆向きに、
飛車が X へ到達可能な中間マス列を数手ぶんブーストする。C-8 より履歴寄りの
アプローチだが、観測列との逐次整合性を保てる（strict のまま修復できる）
利点がある。実装コストは C-8 より軽い可能性が高く、C-8 と並行して
検討する価値がある（同じ問題への異なる角度: C-8=大域の弱整合信念の補完、
多段ガイド=strict 粒子そのものの復活、相手モデル=向かい飛車のような
人間的再配置を事前分布に組み込む、の3系統は互いに独立に価値がある）。

## フェーズ分割（案）

|---|---|---|

| P3 | `synth` 身分の導入と評価側の較正（stratified_sample・p_legal ブレンド、厳密とは別勘定）。全滅時・クリーン粒子極小時の砦として C-7 パイプラインへ統合（総質量上限・p_legal 非混入）。C-7 P3 と同じ Brier/logloss 較正測定を追加（全滅回避だけを合格条件にすると嘘粒子が勝つため） |
| P4 | 配置事前分布の生成モデル向け再フィット（fit_particles/fit_opp の拡張） |
| P5 | シナリオ回帰（keima・ansatsu・kakunari 全体、遂行率）・アリーナ確認（synth 無効との match_seed ペア比較必須） |

## 主要リスク（レビューで特に見てほしい）

1. **負の証拠の時間減衰**（上記の訂正）: 窓の長さ・減衰関数の設計。
   C-7 の REJUV_SNAPSHOTS と同じ「限定的な履歴」への回帰であり、
   「履歴なし」という当初の触れ込みは正確ではない。この前提のズレを
   踏まえた上で C-8 が C-7 に対して持つ本当の優位性（何が軽くなるのか）を
   明確化する必要がある
2. **テンポ収支の会計**: 持ち駒復帰駒（相手が取って再度打った自分の駒）の
   扱い。同じ役割の駒が複数回捕獲・投入された場合、多重集合として役割単位で
   管理すれば個体追跡は不要なはずだが、実装の詰めが必要
3. **合成配置の身分**（synth 層）: 厳密粒子と同格に扱えるかは実測次第。
   较正を誤ると P1 で経験した prior_weight 崩壊の再発リスクがある
4. **駒の相関を表現できるか**: 案A（独立成分）は「利きの形」のような
   同時制約（例: 二枚の駒が連携して利きを作っている）を表現しにくい。
   kakunari の needle 自体が「複数駒の連携」なので、この弱点が本丸に
   刺さらないか要検討
5. **計算コスト**: 案Aの棄却サンプリングは、負の証拠やテンポ制約が
   厳しいと再試行が増える。思考予算内に収める設計（C-7 と同じ時間打ち切り）

## codex 設計レビュー要点（2026-07-21、gpt-5.5、反映済み）

1. 【最優先】負の証拠の時間減衰の訂正は妥当。C-8 の構造的優位性は
   「中間経路を当てなくてよいこと」に限定される（C-7=path-space sampling、
   C-8=state-space synthesis、という用語整理を追加）
2. テンポ収支は役割ごとの**二部マッチング問題**（生え抜き候補×持ち駒復帰
   候補の割当コスト最小化）。単純な役割多重集合の合算では過小/過大評価。
   成り駒の持ち駒復帰は「打つ1手＋成れるゾーンを通過する最短路」が必要
   （「打った1手だけ」は生駒に限れば正しいが成り駒には不十分）
3. 打ち歩詰めは配置レベルでは原則チェック不能（既知のギャップとして記録）
4. `synth` を厳密粒子と同格にしない。p_legal ブレンド・
   `strict_unique_effective_n` から隔離し、用途を玉位置・利き分布・
   候補手の相対評価に限定（P1 較正崩壊の再発防止）
5. 案A単体は kakunari の needle（複数駒連携）に本丸で刺さらない可能性が
   高い。完成形は「案A（構築的初期化）＋短い案B（MCMC仕上げ）」
6. 統合は「クリーン十分なら不使用・全滅/極小時のみ・質量上限・p_legal
   非混入・synth=0 との match_seed 比較必須」まで具体化
7. 代替/補完として「多段ガイド」（C-7 Guide の複数手依存チェーンへの
   拡張）を並走候補に追加。C-8=大域弱整合信念の補完、多段ガイド=strict
   粒子そのものの復活、相手モデル=人間的再配置の事前分布、の3系統は
   独立に価値がある
8. P1/P2/P3 の合格条件を目的直結の指標（真の玉位置一致率・大駒位置一致率・
   重要地点の利き較正・候補手ランキング改善・Brier/logloss）へ強化

## cursor 設計レビュー要点（2026-07-21、agent CLI ask モード）— 費用対効果への根本的疑義

codex とは異なる角度から、**このスコープのまま実装に入るのは早い**という
主旨の指摘。特に codex の「案A単体では needle に刺さらない」という懸念を
さらに一歩進め、「**案A+短い案Bでも刺さらない可能性が高い**」と結論。

1. 【最重要】**A+B が相関を救う条件が2つとも文書に無い**: (a) スコア関数に
   相互作用項（利きの形・大駒の振り直し・重要地点の利き枚数連携）がある、
   (b) 既に相関を持つ種（taint 等）から出発してその相関を壊さず磨く。
   現行スコア（配置事前×テンポ×直近負の証拠）にはどちらも無く、`likelihood.rs`
   の8特徴量には「向かい飛車・大駒の横転・利き連携」が無いので、短い MCMC は
   **因子化分布の近傍を動くだけで needle 形状を発明しない**。「A初期化+短いB」を
   完成形と呼ぶなら、**B のエネルギーに何を入れるか**こそが設計の本体で、
   それが空欄のままでは「因子化サンプルの局所平滑」で終わる
2. 【最重要】**テンポ収支は中終盤でほぼ無力**: 空盤上の admissible 距離は
   飛車角香で遮蔽無視のため極めて甘く、相手手数 N は中盤以降すぐ大きくなり
   無駄手・駒得手のスラックで制約がほぼ常に成立する。D1 は「偽配置を落とす刃」
   ではなく「明らかに不可能な超遠方配置だけ落とすガードレール」でしかない。
   kakunari 70手目前後はまさに「テンポが緩くて何でも通る」領域で、向かい飛車を
   自然に確率づけるどころか**棄却フィルタとして機能しない**可能性が高い
3. 二部マッチングの仕様穴（実装で詰まる点）: ①持ち駒復帰の「打ちマス」は
   固定でなく自由変数（打ち先についての min を取る必要がある）②案Aの構築中の
   貪欲残予算と最終の最適割当が不一致（提案分布がバイアスされ棄却率増加か
   重みが無意味になる）③成り到達は単純距離でなく（位置×成ったか）の拡張状態
   BFS が必要 ④多重捕獲・再投入で `lost_pieces` の時系列と割当スロット数の
   整合を別途ハードで持つ必要
4. **合成盤面のハード制約に「手番側の静的合法局面」が抜けている**:
   手番が自分側なら、直前に指したのは相手なので**合成した相手玉は
   （直近の王手宣言と整合する場合を除き）王手されていてはいけない**。
   これを入れないと「利きの形は綺麗だが相手が直前に指せない局面」が量産される。
   逆に厳密にやると王手の幾何=経路制約の一部が戻ってきて state-space の
   利点が削れる — C-8 が「履歴を捨てられる範囲」の境界として明記が必要
5. **synth 隔離は p_legal 汚染は防ぐが「再発防止」と言うには過大**:
   評価には p_legal 以外に捕獲期待・取られリスク・利き圧力・王手ソルバー・
   ブラインド玉攻めなど多数のチャネルがあり、synth を利き分布・候補手の
   相対評価に使う時点でこれらが嘘の利き枚数で歪みうる。しかも発動条件が
   「strict 極小」で最もてこが大きい局面なので、薄い嘘でも着手が跳ねうる。
   防げるのは p_legal 実効 n の汚染だけで、P3 の合格条件にチャネル別回帰
   （利きBrier・候補ランキング・反則内訳）を「全滅回避」と並べて必須にすべき
6. 【最重要】**費用対効果**: 同じ残差（c42・7i5g+）への対策として、
   C-8 フルより先に安い3手がある — 多段ガイド（strict を保てる、Guide拡張
   程度のコスト）・相手モデル（向かい飛車等への事前確率付与、strict を保てる）・
   評価だけの「盲・大駒/筋」項（ブラインド玉攻めの横展開、コスト最小、
   strict 不要）。実測（玉信念91.8%→ブラインド玉攻めで continue 2/20→13/20）
   はまさにこの安い手が効いた実例。C-8 の価値が出る条件は
   「多段ガイド＋相手モデルでも strict が蘇生せず、かつ評価用の部分信念
   （玉・大駒筋・利き）でも遂行が頭打ちで、そのうえで synth の利き較正が
   実測でプラス」という狭い条件に限られる。現行のフェーズ順（P1テンポ基盤
   →P2合成→P3統合→P4再フィット）は「効くか分からない生成器を先に作り、
   事前分布の再フィットが後」で**生成の質は事前依存なので順序が逆**
7. 見落としリスク: 鶏卵問題（向かい飛車を好まない事前のまま A/B を作っても
   指標が動かず、失敗を「合成アルゴリズム不足」と誤診しやすい）・アリーナ
   中立トラップ（C-7 P3 と同じくbot対戦にkakunari型がほぼ出ない）・
   発動しない砦（「クリーン十分なら不使用」だと thin な soft/taint が
   生きている限り synth が一度も走らない）・棄却サンプリングの予算枯渇
   （C-7既知）・MCMCの見かけの改善（taint種だと玉は当たるが駒は嘘のまま
   利きだけ綺麗になり指標は上がるが着手は悪化しうる）

**この指摘を受けた進め方の判断は、まだ下していない**（次のユーザーとの
対話で決定）。選択肢: (a) 元スコープどおり C-8 フルへ進む、(b) 安価な
フェーズ0（多段ガイド・相手モデル・盲大駒項）を先に試し、それでも
足りない場合だけ C-8 の残り（P2〜P4）に進む、(c) C-8 は P1（テンポ制約・
配置合法性チェッカー）だけを「健全性フィルタ」として残し、生成器
