# SPSAチューニング事前レビュー

調査日: 2026-07-13  
対象: 現行 `estimator` の `strategy::EvalParams` と `bin/tune.rs` のSPSA評価経路

## 結論

現状のままSPSAを開始するのは推奨しない。チューニング結果を歪める可能性が高い問題が4点ある。

1. f+ / f− の評価条件が揃っていない
2. ソフト救済粒子の重みが一部の評価で無視されている
3. 2手読みの静的リスク置換が負の評価値で正しく機能しない
4. 現在のパラメータ化に境界クリップと不感帯がある

少なくともこの4点を修正してからSPSAを実行する。

## 優先度A: SPSA前に修正する項目

### 1. f+ / f− の評価条件を共通化する

`bin/tune.rs` は `TSUITATE_BOOK_RR=1` を設定し、現行 `OpeningBook` の定跡選択を巡回させている。

- `src/bin/tune.rs:128-130`
- `src/opening.rs:141-153`

しかし、既定の基準戦略 `estimator_v5` は凍結版内部の独自 `OpeningBook` を使用し、定跡を毎局ランダムに選択する。

- `src/frozen/estimator_v5.rs:778-782`

推定器の乱数シードと評価時のタイブレーク乱数も、f+とf−で共有されていない。

- `src/estimator.rs:93-107`
- `src/strategy.rs:712-726`

加えて、現行側の定跡巡回カウンタはグローバルなAtomicであり、対局を並列実行するスレッドのスケジューリングによって、どの対局・どのプレイヤーへ各ラインが割り当てられるかが変わる。

- `src/selfplay.rs:383-423`

したがって、現在の実装はコメントにある「共通乱数法」にはなっていない。特に候補側の定跡を `TUNE_CANDIDATE_LINE` で固定した場合、現行側の巡回処理は候補には作用せず、ランダムな凍結版基準だけが残る。

#### 既存ログに見られるノイズ

40局評価の既存ログを集計すると、反復ごとの平均 `|f+ - f−|` は以下だった。

| ログ | 平均 `|f+ - f−|` |
|---|---:|
| `tune-antirush.jsonl` | 0.097 |
| `tune-rush.jsonl` | 0.086 |

40局の独立評価で発生する単純な標本ノイズとほぼ同じ規模である。これは真の信号がないことの証明ではないが、推定勾配のかなりの部分が乱数由来である可能性を示す。

#### 推奨修正

- 対局番号とラン全体のseedから、先後・定跡ライン・候補の推定器seed・基準の推定器seed・タイブレークseedを決定する
- 同じ反復のf+とf−で、まったく同じ対局条件列を再利用する
- 並列実行順に依存しない形で対局コンテキストをファクトリへ渡す
- f+を常に先に評価せず、反復ごとに評価順を入れ替えるかランダム化する
- ランseedと各反復の摂動をログへ記録する

### 2. ソフト救済粒子の重みを全評価経路で統一する

現行評価では、ソフト救済粒子に `0.5^penalty` の重みを付けている。

- `src/strategy.rs:634-645`

駒得、合法率、取られリスクなどはこの重みで加重されている。一方、次の項は重みなしで加算され、粒子数で平均されている。

- 自玉周辺への相手圧力
- 相手玉周辺への攻撃圧力
- 相手持ち駒による王手打ち危険度

該当箇所:

- `src/strategy.rs:1128-1138`

また、王手ソルバーへ粒子を渡す際に重みを捨てて `Vec<&Position>` へ変換している。

- `src/strategy.rs:662-668`
- `src/check.rs:65-69`
- `src/check.rs:170-195`

この状態でSPSAを実行すると、`attack_w`、`pressure_w`、`hand_drop_w` などが本来の棋力ではなく、重み付け漏れを補正する値へ寄る可能性がある。

#### 推奨修正

- 王周辺評価の各和にも粒子重みを掛け、重みの合計で平均する
- `CheckSolver` にも `(position, weight)` を渡し、多数決・仮説投票を加重集計する
- strict粒子と複数penaltyのsoft粒子を混在させた回帰テストを追加する

### 3. 2手読みの静的リスク置換式を修正する

1手読みの最終スコアは次の非線形式になっている。

```text
min(p_legal * gain, gain) - (1 - p_legal) * foul_cost
```

- `src/strategy.rs:1200-1208`

`gain < 0` のときは、合法確率による割引を行わず `gain` 自体を採用する。

一方、2手読みでは静的リスクを足し戻す際に、常に `p_legal` を掛けている。

- `src/strategy.rs:743-764`

```text
score + p_legal * replace_rate * (static_risk + measured_delta)
```

このため、負の候補手では「静的リスクの70%を実測リスクに置き換える」という意図どおりの計算にならない。

例として `p_legal=0.5`、静的リスク2、実測リスク1、その他のgainが0の場合、意図した置換後リスクは `0.3*2 + 0.7*1 = 1.3` だが、現在の式ではスコアは `-2 + 0.5*0.7*(2-1) = -1.65` となり、静的リスクが余分に残る。

置換によってgainの符号が変わる場合はさらに不連続になる。

#### 推奨修正

- `EvalOut` に最終スコアだけでなく、リスクを除いた期待値またはgainの内訳を保持する
- 2手読み後のリスクへ置換してからgainを再構築する
- 再構築したgainに対して、合法率と反則コストを含む最終式をもう一度適用する
- 正のgain、負のgain、置換で符号が変わる場合のテストを追加する

### 4. パラメータの境界クリップと不感帯を解消する

#### `mover_w_check` の不感帯

王手時の着手駒リスクは独立した加算項ではなく、quiet/capturedの重みとの `max` で決まる。

- `src/strategy.rs:1094-1101`

現行値は次の通り。

```text
mover_w_captured = 0.988
mover_w_quiet    = 0.671
mover_w_check    = 0.506
```

したがって中心点付近では `mover_w_check` がquiet/capturedのどちらにも負け、ほぼ評価へ作用しない。SPSAでは他のパラメータと同時に摂動したときだけ閾値を越えるため、非滑らかで解釈しにくい勾配になる。

推奨案:

- `mover_check_extra` のような非負の加算パラメータにする
- または `mover_w_check >= mover_w_quiet/captured` を保証するパラメータ化にする

#### 境界クリップ時の勾配分母

チューナーは摂動点を `[0,1]` へクリップするが、勾配の分母は実際の摂動距離にかかわらず常に `2ck` である。

- `src/bin/tune.rs:208-234`

現行の `camp_scale=0.159` は探索範囲 `[0,3]` に対し、初回摂動が実値で±0.24になる。そのため負方向は最初から0へクリップされ、非対称な評価点を対称差分として扱うことになる。`exposed_known=0.113` と範囲 `[0,1.5]` も同様である。

既存のrushログでも、次の境界到達が確認できる。

- `prior_weight_degen = 0`
- `exposed_known = 0`
- `drop_bias = 0.3`（上限）

推奨案:

- 境界で実際に動いた距離を使う勾配推定へ変更する
- 片側が境界に当たる場合は摂動方向または中心を調整する
- 現在値に対して過大な探索範囲を見直す
- 桁の広い正値パラメータには対数座標も検討する

## 優先度B: 同時に改善したい項目

### 再開ログの整合性を検証する

`resume_state` は、ログ内で最後に見つかった `iter.theta` を無条件で再開点として採用する。

- `src/bin/tune.rs:98-125`

次の設定が以前のランと一致しているか検証しない。

- 基準戦略一覧
- `TUNE_CANDIDATE_LINE`
- `TSUITATE_THINK_BUDGET_MS`
- `ARENA_THREADS`
- josekiの内容
- `EvalParams::SPECS` の名前、順序、範囲
- コードリビジョン

さらに、再開時に `best_score` と `best_params` は既定値へ戻るため、`done.best` は再開後の区間だけを対象とした値になる。

- `src/bin/tune.rs:204-206`

推奨修正:

- startイベントに設定fingerprint、git commit、candidate line、think budget、seedを記録する
- 再開時に一致を検証し、不一致なら明示的な上書き指定なしでは停止する
- best情報とRNG状態、または反復番号から再現できるseed設計を復元する
- 既にdoneがあるログの再利用方針を明確にする

### ログ書き込み失敗を無視しない

現在の `log_line` はserialize、write、flushの失敗をすべて無視する。

- `src/bin/tune.rs:79-85`

Spot VM上でディスク障害等が起きると、再開情報を失ったまま高価な計算を続ける可能性がある。書き込み失敗時は即座にエラー終了させる。

また、再開用thetaは丸めず完全精度で保存し、表示用の丸めと分離するのが望ましい。

### `games_per_eval` と実局数を一致させる

複数基準戦略を指定すると、現在は次の計算で各基準の局数を決める。

```text
per = max(games_per_eval / baselines.len(), 2)
```

- `src/bin/tune.rs:54-76`

除算の余りを捨てるため、表示される局数と実局数が一致しない。基準数が多く指定局数が少ない場合は、実局数が指定値を超える。

余りを各基準へ配分するか、割り切れない指定をエラーにする。また、先後を揃えるため各基準の局数は偶数を保証する。

### 対局内訳をログへ残す

現在のiterログは `f_plus` と `f_minus` の集約スコアしか持たない。次も記録する。

- 勝・敗・引き分け
- `max_plies` 件数
- 反則負け、時間切れ、詰み等の終局理由
- 平均手数
- 候補と基準の平均反則数
- 思考時間の平均・p99・最大
- 基準戦略ごとのスコア

これにより、棋力ではなく引き分け化や時間使用量でスコアが上がったケースを検出できる。

### 200手引き分けによる目的関数のずれを監視する

アリーナは千日手を実装せず、200手到達を引き分けとしている。

- `src/selfplay.rs:23-27`
- `src/selfplay.rs:167-170`

実サービスと終局条件が異なる場合、負けそうな対局を200手まで引き延ばして0.5点を得る方向へパラメータが寄る可能性がある。

まず `max_plies` の割合をチューニングログへ記録する。無視できない割合なら、千日手検出または本番に近い終局条件を実装してからチューニングする。

### 最終中心点を評価する

`done.final` は最後に更新された中心点だが、その中心点自体は評価されていない。評価済みなのは最後のf+ / f−だけである。

完走後に中心点を追加評価し、そのスコアと内訳を `done` へ記録する。最終採用判断は従来どおり独立したCIガントレットで行う。

## テスト状況

調査時の実行結果:

```text
cargo test
  57 passed / 0 failed / 1 ignored

cargo test --release -- --ignored
  shogi::tests::perft_initial_deep passed
```

通常テストと深い将棋ルール検証は成功した。

一方、`bin/tune.rs` にはユニットテストがない。境界摂動、再開互換性、局数配分、ログ復元、決定論的なf+/f−ペアリングについてテストを追加する必要がある。

次の厳格な静的チェックは既存警告により失敗した。

```text
cargo clippy --all-targets --all-features -- -D warnings
```

チューニング関連では、`DEPTH2_TOP_K` と `DEPTH2_PARTICLES` の未使用定数が残っている。

- `src/strategy.rs:191-195`

実装値が `SearchBudget::from_ms` 内に直接書かれており、定数との二重管理になっているため統一する。

## 推奨する修正順

1. 対局コンテキストと乱数seedを導入し、f+ / f−を完全にペアリングする
2. ソフト救済粒子の重みを王周辺評価と王手ソルバーまで一貫させる
3. 2手読み後にgainと最終スコアを再構築する
4. `mover_w_check` のパラメータ化と境界摂動を修正する
5. チューナーの再開検証、完全精度ログ、エラー処理を追加する
6. 対局内訳と `max_plies` をログへ追加する
7. チューナーの回帰テストを追加する
8. 小規模な決定論的スモークランで、同一パラメータ同士の差がほぼ0になることを確認する
9. SPSAを実行する
10. 得られた `done.final` を独立した200局フルガントレットで検証する
